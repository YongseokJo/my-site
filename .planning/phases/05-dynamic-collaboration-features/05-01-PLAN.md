**3. Optimize middleware** `src/middleware.ts` per research Pitfall 2. Add path-based filtering to skip Supabase auth on static pages. Only create the Supabase client and call `getUser()` for routes that need auth:
- Paths that NEED auth: `/dashboard`, `/software/issues`, `/software/proposals`, `/login`, `/signup`, `/pending-approval`, `/api/`
- All other paths: set `context.locals.user = null` and skip Supabase entirely.
- Keep the existing env var check for prerendering.
- For `/software` (exact match, no trailing subpath), still create the client so UserMenu can show logged-in state.

Pattern:
```typescript
const authPaths = ["/dashboard", "/software", "/login", "/signup", "/pending-approval", "/api/"];
const needsAuth = authPaths.some(p => context.url.pathname === p || context.url.pathname.startsWith(p + "/"));
if (!needsAuth) {
  context.locals.user = null;
  return next();
}
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | LoginForm | mitigate | Auth handled by Supabase server-side; passwords never stored client-side; POST to /api/auth/signin validates credentials server-side |
| T-05-02 | Elevation of Privilege | middleware.ts | mitigate | Path-based filtering only SKIPS auth for truly static routes; all /dashboard, /software/issues, /api paths still run full getUser() check |
| T-05-03 | Information Disclosure | pending-approval.astro | accept | Page shows generic "pending approval" message; no user data exposed to unapproved users |
| T-05-04 | Tampering | SignupForm | mitigate | Server-side register endpoint handles signup; client sends FormData over HTTPS; Supabase handles password hashing |
</threat_model>
