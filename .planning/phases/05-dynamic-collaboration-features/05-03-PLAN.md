<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Supabase | All dashboard mutations (approve user, review proposal, assign issue) go through browser client |
| Server -> Supabase | Auth guard in .astro page validates user and fetches role |
| Role prop -> Island | Role determined server-side, passed as prop; client cannot modify the server check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-10 | Elevation of Privilege | Dashboard.tsx admin panel | mitigate | RLS policy "Admins update any profile" ensures only admin role can approve users; UI check is cosmetic, DB enforces |
| T-05-11 | Elevation of Privilege | Dashboard.tsx proposal review | mitigate | RLS policy "PI and admin update proposals" restricts proposal status updates to admin/pi_mentor only |
| T-05-12 | Spoofing | Dashboard.tsx role prop | mitigate | Role prop is set server-side in dashboard.astro from profiles table query; client-side role is display-only, all mutations go through RLS |
| T-05-13 | Tampering | Dashboard.tsx issue assignment | mitigate | RLS policy "Admins update any issue" restricts issue updates to admins and reporters; assignee must reference valid profiles(id) via FK |
| T-05-14 | Elevation of Privilege | Unapproved user access | mitigate | Server-side check in dashboard.astro verifies profile.approved before rendering; RLS is_approved() function blocks all data queries for unapproved users |
| T-05-15 | Information Disclosure | Viewer role data access | accept | Viewers see approved proposals only (RLS enforced); no issue access. This is by design per D-13 |
</threat_model>
